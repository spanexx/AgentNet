### Logs confirmed

- `requests.jsonl` captures every request/response (append-only audit log, NOT used for learning).
- `usageCount` increments on every `/api/search` call that returns matching messages (confirmed: first call shows `usage: 0`, second call shows `usage: 1`).

### How the system learns today

**Semantic learning (active only when embeddings work)**
- `normalizeMessage()` → searches semantic clusters → updates centroids/density/frequency via `updateCentroid` + `maybeDiscover`
- Currently: `providers_used: []`, `source: "fallback"` — no real embedding provider is configured, so semantic learning is bypassed
- To enable: set `OPENAI_API_KEY` or `GOOGLE_API_KEY`, or run local sidecar on `127.0.0.1:8100`

**Retrieval "learning" (always active)**
- `usageCount` and `lastUsedAt` are updated on every search result set via `updateMany`
- This is coarse — it counts every search impression, not confirmed reuse

### Open issues

1. **No working embedding provider** — semantic cluster evolution is frozen; fix API keys or sidecar to resume centroid learning
2. **Usage tracking is impression-based, not confirmation-based** — `usageCount` increments on every search hit, not when a client explicitly marks a message as reused; consider adding a `POST /api/message/:id/use` endpoint
3. **Fallback normalization** — with no embedding provider, all normalization falls back to `source: "fallback"` with hardcoded `frontend_dashboard` intent; real SDG hypothesis diversity is absent until embeddings are restored
